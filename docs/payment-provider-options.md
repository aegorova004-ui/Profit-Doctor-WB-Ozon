# Payment provider options

Обновлено: 21 июля 2026 года.

Цель документа — выбрать реалистичный путь к оплате тарифов Profit Doctor в рублях для пользователей из России и СНГ. Это не интеграция и не юридическое заключение: перед подключением нужно подтвердить договор, тарифы, доступность для выбранной формы бизнеса и требования 54-ФЗ в личном кабинете провайдера.

## Короткое решение

Для MVP стоит начинать с hosted payment form у провайдера, а не со своей платёжной формы. Так мы быстрее получим оплату тарифа, меньше касаемся карточных данных и можем сосредоточиться на webhook-подтверждении статуса платежа.

Приоритет проверки:

1. **ЮKassa** — основной кандидат для первого подключения.
2. **Т-Банк Интернет-эквайринг** — сильная альтернатива, особенно если расчётный счёт или бизнес-процессы уже в Т-Банке.
3. **Robokassa** — запасной кандидат, если нужна максимально простая платёжная форма/модули.
4. **CloudPayments** — кандидат для будущих рекуррентных сценариев, если условия подключения и поддержка РФ подходят бизнесу.

## Минимальный технический контракт

Будущая интеграция должна быть server-only:

- client получает только ссылку/redirect на оплату или hosted widget params без секретов;
- секреты провайдера хранятся только в env/secrets;
- webhook проверяет подпись/токен провайдера до изменения тарифа;
- повторный webhook идемпотентен по `providerPaymentId` и внутреннему `orderId`;
- сумма хранится в integer-копейках, валюта явно `RUB`;
- статус подписки меняется только после подтверждённого successful/captured payment event;
- возвраты и отмены не удаляют историю, а создают отдельное событие;
- email покупателя нормализуется: trim + lowercase до записи;
- сырые webhook payload не логируются целиком.

## Минимальные env-переменные

Имена предварительные, финальные зависят от выбранного провайдера:

```bash
PAYMENT_PROVIDER="disabled"
PAYMENT_RETURN_URL=""
PAYMENT_WEBHOOK_SECRET=""
PAYMENT_SHOP_ID=""
PAYMENT_API_KEY=""
```

Для локальной разработки первый режим должен оставаться `disabled` или `mock`, чтобы случайно не создать реальный платёж.

## Что не делаем в MVP

- Не собираем и не храним карточные данные в Profit Doctor.
- Не включаем оплату без webhook-подтверждения на сервере.
- Не активируем тариф только по redirect на success page.
- Не обещаем подписки/автоплатежи, пока не выбран провайдер и не проверены условия рекуррентных платежей.

# Форматы отчётов

MVP принимает `.xlsx` и `.csv`, но поддержка определяется не расширением, а версией адаптера. Актуальные названия колонок WB/Ozon нельзя угадывать или закреплять без проверенного образца и fixture.

## Поток импорта

1. Проверить расширение, MIME-тип, размер и сигнатуру файла.
2. Вычислить технический checksum для идемпотентности; не выводить его как идентификатор пользователя.
3. Прочитать заголовки и ограниченную выборку строк без записи содержимого в логи.
4. Определить маркетплейс и версию формата по набору обязательных признаков.
5. Преобразовать строки в канонический контракт.
6. Провести валидацию типов, знаков, валюты, периода и SKU.
7. Сохранить ошибки импорта и происхождение полей; только после успешной валидации запускать расчёты.

Неизвестный набор колонок должен завершиться понятной ошибкой и предложением ручного маппинга, а не молчаливым частичным импортом.

## Каноническая строка

Минимальный интерфейс нормализованной операции:

```ts
type Marketplace = "wildberries" | "ozon";

interface NormalizedReportRow {
  sourceRowNumber: number;
  sku: string;
  offerId: string | null;
  productName: string | null;
  quantity: number;
  revenueKopecks: number | null;
  marketplaceCommissionKopecks: number | null;
  logisticsKopecks: number | null;
  storageKopecks: number | null;
  returnsKopecks: number | null;
  penaltiesKopecks: number | null;
  advertisingKopecks: number | null;
  costOfGoodsKopecks: number | null;
  otherExpensesKopecks: number | null;
}
```

Маркетплейс, `formatVersion` и количество исходных операций `sourceRowCount` хранятся в результате парсинга. Денежные значения — безопасные integer-копейки или `null`, если исходного компонента нет. Расходы в каноническом контракте неотрицательны; направление исходной операции обрабатывает адаптер. Неизвестное значение нельзя тихо заменить нулём: `null` передаётся в финансовое ядро и делает результат оценкой.

## Версии адаптеров

Каждый подтверждённый формат получает отдельный идентификатор, например `wb:<report-type>:YYYY-MM` или `ozon:<report-type>:YYYY-MM`. Для версии фиксируются:

- тип и источник выгрузки;
- обязательные и необязательные заголовки;
- правила знаков, возвратов и корректировок;
- единицы, валюта, формат дат и локаль числа;
- fixture без реальных коммерческих данных;
- ожидаемый нормализованный результат и контрольные суммы.

Автоопределение не должно выбирать адаптер, если подходят несколько версий. В этом случае импорт останавливается и просит пользователя уточнить тип отчёта или сопоставить колонки.

## Wildberries

Первый preview-вариант имеет идентификатор `wb:api-financial-report:preview-2026-07`. Он нужен для проверки архитектуры импорта и не считается подтверждённой поддержкой выгрузки из кабинета. Production-вариант выбирается после получения актуального анонимизированного отчёта. Нужно подтвердить колонки для SKU/артикула продавца, операции, даты, количества, розничной суммы, комиссии, логистики, хранения, штрафов и других удержаний.

Синтетический образец `tests/fixtures/reports/wb-financial-report-api-synthetic.xlsx` воспроизводит 22 поля из официальной документации WB API: период отчёта, валюту, идентификаторы операции и товара, штрихкод, тип документа, количество, розничные суммы, сумму к выплате, логистику, штрафы, доплаты, хранение и удержания. В нём есть продажа и возврат, но нет реальных данных кабинета; поэтому образец подходит для тестирования XLSX-импорта и распознавания колонок, но не подтверждает фактический формат выгрузки из кабинета продавца.

Preview-адаптер:

- ищет строку заголовков среди первых 20 строк и требует 14 финансовых полей;
- принимает только `RUB`/`RUR`, операции `Продажа` и `Возврат`, не более 100 000 операций и 100 колонок;
- читает числа как десятичные строки и переводит рубли в копейки через `BigInt`, не используя JS float;
- агрегирует продажи и возвраты по `barcode`, использует `nm_id` как идентификатор WB;
- вычисляет комиссию как разницу `retail_amount − ppvz_for_pay`, отдельно учитывает логистику, хранение, штрафы, удержания и возвраты;
- сверяет итог каждого SKU с суммой `for_pay` с допуском в одну копейку;
- оставляет рекламу и себестоимость как `null`, поэтому результат всегда явно помечен оценкой.

После нормализации пользователь может вручную задать себестоимость единицы по `sku`. Это значение не изменяет распознанный файл: слой анализа помечает источник как `user-unit-cost`, умножает цену на положительное итоговое количество и передаёт общую себестоимость в финансовое ядро. В UI рядом с каждой суммой показываются понятное название исходного поля или формулы и статус отсутствующих данных.

Неподдерживаемая валюта, операция, знак, дублированная колонка, конфликт идентификаторов и несошедшаяся выплата останавливают импорт. Публичная ошибка содержит код, поле и при возможности номер строки, но не полную исходную строку.

Особое внимание: сторно, возвраты, корректировки продаж, несколько строк одной операции и возможное отсутствие рекламы в финансовом отчёте.

## Ozon

Первый поддерживаемый вариант также закрепляется только по актуальному fixture. Нужно подтвердить идентификатор товара, тип начисления, дату, количество, сумму продажи, комиссию, логистику, хранение, возвраты и прочие услуги.

Особое внимание: разнесённые начисления по одной продаже, услуги отдельными строками, возвраты и разные источники рекламы.

## Ошибки и приватность

Ошибка содержит код, номер строки и публичное имя поля, но не полную исходную строку. Файл ограничивается по размеру и количеству строк; макросы и исполняемое содержимое не запускаются. Fixtures должны быть синтетическими либо необратимо анонимизированными.

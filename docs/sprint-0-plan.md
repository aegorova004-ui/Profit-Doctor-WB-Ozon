# Sprint 0: Foundation

План создан 13 июля 2026 года. Статус: **локальная реализация завершена, нужна внешняя проверка**. Цель — получить воспроизводимую, проверяемую основу для первого полезного импорта без production-обещаний.

## Результат спринта

Разработчик устанавливает проект с нуля, запускает его и видит базовые экраны; Prisma schema и финансовое ядро проверяются автоматически; GitHub ведёт CI, недельное планирование и ежедневные отчёты.

## Работы и acceptance criteria

### 1. Инструменты и приложение

- [x] Создан Next.js 16 + TypeScript проект на npm.
- [x] Есть скрипты `format:check`, `typecheck`, `lint`, `test`, `build`, `db:generate`, `db:validate`; каждый завершается с кодом `0` после чистого `npm ci`.
- [x] `.env.example` перечисляет обязательные переменные без секретов; `.env*` исключены из Git, кроме примера.
- [x] `npm run dev` открывает приложение без runtime-ошибки.

### 2. База данных

- [ ] PostgreSQL подключается через `DATABASE_URL`.
- [x] Prisma schema содержит `User`, `UploadedReport`, `ReportRow`, `ProductProfitSnapshot`, `SubscriptionPlan` и связи принадлежности.
- [x] Суммы отчётов и снимков в БД используют `Decimal(18,2)`, цена тарифа — integer-копейки; уникальные ограничения защищают от очевидных дублей, индексы покрывают владельца, отчёт и SKU.
- [x] `npm run db:generate` и `npm run db:validate` проходят с заданным `DATABASE_URL`.

### 3. Первый интерфейс

- [x] Landing показывает оффер «Загрузите отчёт WB/Ozon — за 3 минуты покажем товары, которые съедают вашу прибыль» и CTA загрузки/demo.
- [x] Auth skeleton явно сообщает, что аккаунт и magic link ещё не создаются.
- [x] Upload page принимает выбор `.xlsx`/`.csv`, показывает ограничения и состояния загрузки/ошибки, но не отправляет файл.
- [x] Семантические подписи полей доступны; при ширине 390 px нет горизонтального переполнения и основной CTA виден.

### 4. Доменное ядро и импорт

- [x] Определён интерфейс парсера и каноническая нормализованная строка; версионирование адаптеров зафиксировано в документации Sprint 1.
- [x] Добавлены синтетические минимальные fixtures WB/Ozon без реальных данных продавцов.
- [x] Финансовое ядро принимает и проверяет безопасные integer-копейки, а суммы не считает через JS float.
- [x] Unit-тесты покрывают прибыль/убыток, нулевые знаменатели маржи/ROI, неполные данные, точность до копейки и переполнение.

### 5. Документация и GitHub

- [x] Созданы `AGENTS.md`, `README.md`, `PROJECT_STATE.md`, `ROADMAP.md`, `CHANGELOG.md` и документы в `docs/`.
- [x] Добавлены PR/Issue templates и workflow для CI, недельного спринта и будничного отчёта.
- [ ] CI успешно проходит в GitHub; scheduled workflows имеют разрешение `issues: write` и ручной запуск не создаёт дубль за ту же московскую дату.
- [x] Первый setup оформлен в отдельной ветке `codex/sprint-0-foundation`; локальный коммит создан после финального self-review.

## Проверка спринта

```bash
npm ci
npm run db:generate
npm run db:validate
npm run format:check
npm run typecheck
npm run lint
npm test
npm run build
```

Дополнительно вручную: открыть landing и upload page на desktop/mobile, проверить отсутствие чувствительных данных в логах, вручную запустить оба scheduled workflow и повторить запуск для проверки дедупликации.

## Риски и границы

- Адаптеры Sprint 0 задают контракт, но не обещают поддержку реальной выгрузки до сверки с актуальными анонимизированными fixtures.
- Auth skeleton не равен production-аутентификации.
- CI не поднимает production-БД и не выполняет deploy.
- Платежи, dashboard и экспорт относятся к следующим спринтам.

## Следующая задача после приёмки

Реализовать импорт одного подтверждённого формата Wildberries и сверить итог по SKU с ручным контрольным расчётом.

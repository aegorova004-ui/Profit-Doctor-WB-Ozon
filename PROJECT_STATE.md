# Состояние проекта

Обновлено: 14 июля 2026 года.

## Текущая стадия

`Sprint 1: Первый полезный импорт` — Sprint 0 принят и объединён с `main`; реализованы локальный preview-импорт синтетического WB XLSX, демо-отчёт, ручной пересчёт с себестоимостью, диагностический слой и локальный CSV/XLSX-экспорт. Подтверждение фактической выгрузки ещё впереди.

## Уже работает

- Создан базовый проект Next.js 16 с TypeScript, React 19 и Tailwind CSS.
- Landing, `/login` и `/upload` рендерятся; выбор файла проверяет расширение, MIME-тип, пустой файл и лимит 10 МБ.
- `/upload` локально читает синтетический WB XLSX, распознаёт preview-версию, агрегирует 4 операции в 3 SKU, сверяет `for_pay` и показывает сводку без сетевых запросов.
- На `/upload` есть кнопка «Открыть демо-отчёт»: она загружает публичный синтетический XLSX из `public/demo/wb-demo-report.xlsx`, разбирает его в браузере и позволяет быстро повторить основной сценарий без ручного выбора файла.
- После импорта пользователь вводит себестоимость единицы для каждого SKU; приложение точно умножает её на количество, пересчитывает результат до рекламы и не смешивает пустое поле с введённым нулём.
- Таблица SKU раскрывает источники выручки и каждого удержания, показывает происхождение себестоимости и явно отмечает «Убыток», «Пока в плюсе» или «Нужна себестоимость».
- Блок «Диагноз Profit Doctor» ранжирует убыточные SKU, показывает разрыв до безубыточности, предельную себестоимость до рекламы, положительный запас и крупнейшую известную статью расходов.
- Результат можно скачать в CSV или XLSX без серверной отправки: экспорт сохраняет версии формата и формулы, источники, отсутствующие поля, диагноз и рекомендации.
- Денежные значения XLSX читаются как строки и переводятся в integer-копейки через `BigInt`; комиссия, логистика, хранение, возвраты, штрафы и удержания имеют отдельные поля.
- Неизвестная валюта, операция, знак или несошедшаяся выплата останавливают импорт с публичной ошибкой без содержимого полной строки.
- На landing выровнены карточки аудитории, увеличена мелкая типографика, формула прибыли оформлена как уравнение, а тарифы показывают наследование возможностей и пояснения к каждому пункту.
- Демо-диагностика на первом экране отрисовывается цельным PNG; оригинальный пиксельный маскот Profit Doctor появляется в самой карточке и отдельном блоке-экскурсии, где держит руку поднятой при наведении или кратко поднимает её после тапа.
- Информационные подсказки тарифов открываются наведением на десктопе и нажатием на сенсорных устройствах; активная карточка поднимается поверх соседних.
- Prisma schema и начальная SQL-миграция содержат `User`, `UploadedReport`, `ReportRow`, `ProductProfitSnapshot` и `SubscriptionPlan`.
- Финансовое ядро считает суммы в integer-копейках, помечает неполный результат как оценку и возвращает маржу/ROI в базисных пунктах.
- Контракт парсера, автоопределение WB/Ozon по заголовкам и синтетические fixtures готовы как основа Sprint 1.
- Добавлен синтетический XLSX-fixture финансового отчёта Wildberries: 22 API-поля, продажа, возврат, логистика и хранение. Файл не содержит персональных или коммерческих данных и не заменяет актуальную анонимизированную выгрузку продавца.
- Чистый `npm audit` проходит без известных уязвимостей; unit- и integration-тесты покрывают импорт, демо-запуск, точный пересчёт себестоимости, убыточные SKU и диагностику безубыточности, маршруты `/`, `/login`, `/upload` собираются статически.
- Подготовлены проектные инструкции, план Sprint 0, release checklist, шаблоны GitHub и CI-workflow.
- Настроены автоматические GitHub Issues для недельного планирования и вечерних отчётов.
- Remote `origin` подключён к `aegorova004-ui/Profit-Doctor-WB-Ozon`, Issues включены, PR #2 прошёл CI и объединён с `main`.
- Главный оффер и внешний вид страниц подтверждены пользователем.

## Пока не работает

- Сквозной сценарий: вход → подтверждённый импорт → себестоимость → точная прибыль → экспорт.
- Парсинг подтверждённых реальных форматов Wildberries и Ozon; текущая WB-версия имеет статус preview.
- Авторизация, тарифные ограничения, платежи и админ-активация.
- Применение миграции к реальной PostgreSQL: в текущем окружении сервер БД не подключён.
- Production-деплой.

## Сломано

Подтверждённых дефектов нет. Продукт ещё не готов к реальным отчётам: auth остаётся прототипом, а WB-адаптер проверен только на синтетическом API-подобном XLSX.

## Ближайшие задачи

1. Получить полностью анонимизированный актуальный отчёт Wildberries и сравнить его с preview-контрактом. Переданный 13 июля XLSX — явно синтетический тестовый файл и этот пункт не закрывает.
2. Проверить большой файл, мобильную таблицу и основной сценарий в браузере после демо-запуска.
3. Подключить тестовую PostgreSQL и применить начальную миграцию через `npm run db:migrate`.
4. Вручную проверить дедупликацию двух issue-workflow повторным запуском в ту же московскую дату.

## Известные риски

- Форматы и названия колонок WB/Ozon меняются; каждый поддерживаемый вариант нужен как отдельная версия контракта и fixture.
- Без себестоимости и части удержаний прибыль можно показывать только как оценку.
- Реальные отчёты содержат коммерчески чувствительные данные; для тестов допустимы только синтетические или анонимизированные файлы.
- GitHub Actions создадут Issues только после включения Issues и публикации workflow в default branch; workflow сами запрашивают минимальное право `issues: write`.
- Текст внутри PNG-диагностики нельзя обновить через HTML или локализацию: при изменении содержания потребуется перегенерация растрового ассета.

## Запуск

```bash
npm run db:generate
npm run db:validate
npm run dev
```

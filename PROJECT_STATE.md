# Состояние проекта

Обновлено: 13 июля 2026 года.

## Текущая стадия

`Sprint 0: Foundation` — локальная реализация завершена, осталась внешняя проверка PostgreSQL и GitHub Actions.

## Уже работает

- Создан базовый проект Next.js 16 с TypeScript, React 19 и Tailwind CSS.
- Landing, `/login` и `/upload` рендерятся; выбор файла проверяет расширение, MIME-тип, пустой файл и лимит 10 МБ без отправки на сервер.
- На landing выровнены карточки аудитории, увеличена мелкая типографика, формула прибыли оформлена как уравнение, а тарифы показывают наследование возможностей и пояснения к каждому пункту.
- Оригинальный маскот Profit Doctor появляется в демо-диагностике и отдельном блоке-экскурсии от убыточного SKU к решению.
- Prisma schema и начальная SQL-миграция содержат `User`, `UploadedReport`, `ReportRow`, `ProductProfitSnapshot` и `SubscriptionPlan`.
- Финансовое ядро считает суммы в integer-копейках, помечает неполный результат как оценку и возвращает маржу/ROI в базисных пунктах.
- Контракт парсера, автоопределение WB/Ozon по заголовкам и синтетические fixtures готовы как основа Sprint 1.
- Чистый `npm ci`, `npm audit` и полный `npm run check` проходят; 21 unit-тест зелёный, маршруты `/`, `/login`, `/upload` собираются статически.
- Подготовлены проектные инструкции, план Sprint 0, release checklist, шаблоны GitHub и CI-workflow.
- Настроены автоматические GitHub Issues для недельного планирования и вечерних отчётов.

## Пока не работает

- Сквозной сценарий: вход → загрузка отчёта → расчёт → таблица SKU → экспорт.
- Парсинг реальных форматов Wildberries и Ozon.
- Авторизация, тарифные ограничения, платежи и админ-активация.
- Применение миграции к реальной PostgreSQL: в текущем окружении сервер БД не подключён.
- Production-деплой.

## Сломано

Подтверждённых дефектов нет. Продукт ещё не готов к реальным отчётам: upload и auth остаются честными прототипами без серверной обработки.

## Ближайшие задачи

1. Опубликовать setup-ветку и проверить CI и два issue-workflow в GitHub.
2. Подключить тестовую PostgreSQL и применить начальную миграцию через `npm run db:migrate`.
3. Получить полностью анонимизированный актуальный отчёт Wildberries и зафиксировать первую версию формата.
4. Реализовать серверную загрузку, нормализацию и первый полезный импорт Wildberries.

## Известные риски

- Форматы и названия колонок WB/Ozon меняются; каждый поддерживаемый вариант нужен как отдельная версия контракта и fixture.
- Без себестоимости и части удержаний прибыль можно показывать только как оценку.
- Реальные отчёты содержат коммерчески чувствительные данные; для тестов допустимы только синтетические или анонимизированные файлы.
- GitHub Actions создадут Issues только после включения Issues и публикации workflow в default branch; workflow сами запрашивают минимальное право `issues: write`.

## Запуск

```bash
npm ci
npm run db:migrate
npm run db:generate
npm run db:validate
npm run dev
```

Нужен локальный `.env` с `DATABASE_URL`; точные переменные перечисляются в `.env.example`.


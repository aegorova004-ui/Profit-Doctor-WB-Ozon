# Инструкция для агентов

## Цель и главный сценарий

`Profit Doctor WB/Ozon` — web-SaaS для продавцов из России и СНГ. Главный сценарий: пользователь входит в сервис, загружает XLSX/CSV-отчёт Wildberries или Ozon, видит прибыльность и проблемные SKU, затем скачивает результат.

Каждая задача должна улучшать точность расчётов, стабильность импорта, пользовательский опыт, продажи, удержание или объём ручной работы. Не усложняйте стек «на будущее».

## Контекст и режим работы

1. Сначала прочитайте `PROJECT_STATE.md`, затем релевантные документы и код. Для Next.js 16 перед изменением кода откройте подходящий раздел в `node_modules/next/dist/docs/`.
2. Классифицируйте задачу: `simple`, `standard`, `complex` или `critical`. Используйте соответственно самый быстрый, сбалансированный, сильный или максимальный доступный режим. Максимальный reasoning нужен только для риска денег, данных, безопасности, auth или платежей.
3. Не читайте весь репозиторий без причины. Ищите точечно через `rg`; для большой задачи сначала составьте короткий план, после изменений проведите self-review.
4. До работы кратко зафиксируйте тип задачи, режим, файлы и проверки. После работы перечислите изменения, проверки, риски, пользовательскую проверку и следующий шаг.

## Приоритеты

1. Обратная связь пользователя.
2. Блокирующие баги.
3. Ошибки финансовых расчётов.
4. Безопасность и приватность.
5. Оплата, конверсия и удержание.
6. UX, затем рефакторинг и косметика.

Если прошлый спринт не закончен, сначала перенесите и завершите его критичные задачи.

## Разработка и GitHub

- Одна задача — одна ветка `codex/<краткое-имя>`; одна логичная фича или исправление — один PR.
- В PR укажите причину, проверку и риски; для рискованных изменений добавьте rollback plan. Не объединяйте PR при красном CI или критичных замечаниях.
- В понедельник проверьте состояние, незавершённые задачи, баги, обратную связь и CI; затем составьте недельный план с задачей, влияющей на деньги, конверсию или удержание.
- С понедельника по пятницу завершайте день кратким отчётом: сделано, проверки, что проверить пользователю, блокеры, переносы и следующий шаг.
- Автоматические напоминания создают `.github/workflows/weekly-sprint.yml` и `.github/workflows/daily-report-reminder.yml`.

## Проверки

- Перед PR запускайте как минимум `npm run typecheck`, `npm run lint`, `npm test` и `npm run build`; также `npm run format:check`, `npm run db:generate` и `npm run db:validate`, если затронуты форматирование или Prisma.
- Финансовые формулы покрывайте unit-тестами, парсеры — fixtures и parser-тестами, загрузку — integration-тестом. Критичный сценарий покрывайте e2e smoke-тестом.
- Не скрывайте пропущенные или упавшие проверки. Не меняйте финансовые формулы без тестов и обновления `docs/finance-formulas.md`.

## Деньги, данные и безопасность

- Нормализатор переводит денежные значения в целые безопасные integer-копейки; Prisma хранит `Decimal(18,2)`. Не считайте деньги через JS float.
- Маржа и ROI — доли; при нулевом знаменателе возвращайте `null`/`undefined`, а не бесконечность или ноль. Показывайте источники полей, допущения и статус «Оценка», если данных не хватает.
- Не логируйте отчёты целиком, токены, ключи и персональные данные. Секреты хранятся только в env/secrets; `.env` не коммитится.
- Перед записью обрезайте пробелы в email и приводите его к нижнему регистру; `citext` в PostgreSQL дополнительно защищает регистронезависимую уникальность.
- Проверяйте владельца данных, размер, MIME-тип и расширение загрузки. Платёжные события подтверждайте на сервере.

## Ограничения продукта

Публикация и продажи должны быть доступны из России. Основной канал — web-SaaS, оплата — в рублях через доступный российский эквайринг; Telegram/VK подходят для уведомлений и маркетинга. Не стройте MVP вокруг зарубежного банковского счёта, App Store, Google Play или Steam.
